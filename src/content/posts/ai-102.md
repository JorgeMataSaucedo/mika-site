Estudiando para AI-102 me topé con esta pregunta en el practice exam:

> *"Your generative AI solution must remain available if the primary model deployment becomes unavailable. What should you implement?"*

La respuesta oficial era *"Provision a secondary deployment in a different region."* Cerré el libro y me serví otro café. Porque ese consejo te protege contra una caída de región — pero **no contra que el vendor suba precios, cambie términos de uso, o deprecie el modelo que estabas usando.** Y todos esos riesgos son más comunes que una caída de región.

Ese fue el momento en que entendí que la cert no era el techo. Era el **piso mínimo profesional**. Lo que yo necesitaba construir era *eso + un poquito más*. Y "ese poquito más" se convirtió en Mika.

## Lo que vive dentro de Mika (con código real)

### 1 · Diez músicos, un director

Mika habla con **diez proveedores de LLM** detrás de la misma interfaz: OpenAI, Anthropic, Gemini, DeepSeek, Mistral, Qwen, GLM, Kimi, Llama (via Groq), y OpenRouter como agregador. El factory pattern vive en `api/llm_runner.py`:

```python
_PROVIDER_FACTORIES: dict[str, type[ILLMProvider]] = {
    "Anthropic":  AnthropicProvider,
    "OpenAI":     OpenAIProvider,
    "Gemini":     GeminiProvider,
    "DeepSeek":   DeepSeekProvider,
    "Mistral":    MistralProvider,
    "Qwen":       QwenProvider,
    "Glm":        GlmProvider,
    "Kimi":       KimiProvider,
    "Groq":       LlamaProvider,
    "OpenRouter": OpenRouterProvider,
}
```

Cambiar de vendor para una Operation = `UPDATE DataSource SET Provider = 'Mistral' WHERE Alias = 'sandbox-deepseek-llm'`. No CI, no redeploy.

### 2 · Failover con clasificación de errores

La parte que el examen no examina: **no todo error amerita failover.**

```python
def _is_failover_eligible(err: BaseException) -> bool:
    if isinstance(err, ProviderError):
        if err.retryable:                       # rate limit, timeout, 5xx
            return True
        msg = str(err).lower()
        if any(sig in msg for sig in (
            "quota", "payment", "rate", "404",
            "deprecated", "decommissioned",
        )):
            return True
    # InvalidPromptTemplate, UnsupportedProvider, 401 auth — NO failover
    return False
```

Si el prompt está mal formateado, brincar al siguiente vendor **esconde el bug**, no lo resuelve. Lo correcto es **re-raise** y que el ingeniero lo arregle. Si el vendor está caído, brincar **es** la respuesta. La distinción la hace `_is_failover_eligible` y queda registrada en `routing_path`:

```
primary:Anthropic:claude-sonnet-4-6:failed-RateLimitError
fallback0:OpenRouter:anthropic/claude-sonnet-4.6:ok
```

En 6 meses puedes preguntar: *"¿cuántas veces fallamos a primario este mes y a qué fallback caímos?"* — y la respuesta es un query SQL de 10 líneas.

### 3 · Cero API keys en archivos

El examen AI-102 te examina esto y la mayoría de productos AI que veo en GitHub lo reprueban. El secreto **nunca toca disco** — lo resuelve Key Vault con Managed Identity:

```python
class KeyVaultSecretResolver:
    def __init__(self, *, vault_url: str) -> None:
        self._credential = DefaultAzureCredential()
        self._client = SecretClient(vault_url=vault_url, credential=self._credential)

    async def get_secret(self, name: str) -> str:
        secret = await self._client.get_secret(name)
        return secret.value or ""
```

`DefaultAzureCredential` funciona igual en local (con `az login`) que en producción (con la Managed Identity del App Service). Si rotas el secreto, el siguiente request usa el nuevo valor. **Nunca hubo una API key que rotar manualmente.**

### 4 · El cuaderno de Tesorería

Cada llamada deja una fila en `ExecutionLog`:

```sql
CREATE TABLE ExecutionLog (
    ExecutionLogId    BIGINT IDENTITY,
    CorrelationId     UNIQUEIDENTIFIER,    -- trazable end-to-end
    OperationId       INT,
    PrincipalId       INT,                 -- quién lo invocó (RBAC)
    Status            NVARCHAR(20),
    StartedUtc        DATETIME2(7),
    FinishedUtc       DATETIME2(7),
    LlmProvider       NVARCHAR(50),
    LlmModel          NVARCHAR(100),
    LlmInputTokens    INT,
    LlmOutputTokens   INT,
    RoutingPath       NVARCHAR(2000)
);
```

Cuando un cliente pregunta *"¿cuánto me costó la IA este mes, desglosada por vendor?"* — es un `GROUP BY LlmProvider` y listo. No App Insights, no Datadog, no Kusto. **Está en SQL, fila por fila, desde el primer commit.**

## La pregunta que importa

No es *"¿qué modelo usas?"*. Es:

> **"Si en 6 meses un cliente te pregunta de dónde salió este reporte de abril, ¿en cuántos minutos puedes generarle la respuesta?"**

Si tu respuesta es *"déjame revisar logs"* — perdiste el contrato. Si tu respuesta es *"dame el CorrelationId y te lo traigo en 5 minutos"* — ganaste.

Eso es responsible AI con propiedad de verdad. Eso es lo que **AI-102 examina si lo aplicas en serio.** Y eso es lo que la mayoría de "AI startups" no construyen — porque "es mucho trabajo y nadie lo pide".

Lo construí porque trabajando en empresas con auditorías reales aprendí que **si no puedes mostrar quién hizo qué cuándo, estás de la fregada**.

## El moat

Cuando me preguntan *"¿pero qué te impide a Anthropic, OpenAI o Microsoft hacer lo mismo y enterrarte?"*, mi respuesta honest:

**Lo que pueden copiar fácil:** la UI, el soporte multi-proveedor, el concepto de Operations.

**Lo que es muy difícil de copiar:**

- 🟢 **El catálogo SQL como fuente de verdad** — cambia su modelo de negocio (ellos quieren vendor lock-in, no portabilidad)
- 🟢 **El audit trail con triggers en SQL Server** — requiere integrarse con bases legacy del cliente, ellos quieren que migres a la suya
- 🟢 **La filosofía audit-honest by default** — va contra la cultura de "AI es magia, no preguntes"
- 🟢 **Compatibilidad con el ecosistema Azure existente del cliente** — cada uno de ellos quiere que uses SU plataforma

**El moat real no es código. Es alineamiento de intereses con el cliente.** Mika está construida para servir al cliente, no para amarrarlo.
